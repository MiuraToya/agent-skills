# Deterministic Test Engineering

## 概要

テストの品質や書き方にばらつきが出ないように、共通の手順で自動テストを設計・実装する。
このファイルにはどの言語でも使える進め方だけを書き、言語やフレームワークごとの具体的なやり方は `references/` の対応ファイルを参照する。

## 実行ワークフロー

1. 何をどこまでテストするかを決める。
- 対象レイヤー（unit/integration/e2e）をはっきりさせる。
- ビルド設定・テスト設定・既存テストの前提を確認する。
- ツールが入っている前提で進めず、必要なら導入手順を書く。

2. テスト戦略を選ぶ。
- 基本はテストピラミッドを採用する。
- ビジネスロジックが薄く、統合テストで多くのケースをカバーできる場合に限りテストダイヤモンドを採用する。
- 判断に迷うときは `references/strategy-and-layout.md` を参照する。

3. テストの配置を設計する。
- パス表記は常に `/` 区切りで統一する。
- レイヤーごとにディレクトリを分け、要件とテスト対象の振る舞いを1対1で対応付ける。
- 具体的な配置例は `references/strategy-and-layout.md` を参照する。

4. 振る舞いベースでテストを書く。
- private 実装を直接検証せず、公開インターフェース経由で確認する。
- テスト名は `<条件>_<期待結果>` の形で意味が伝わるように書く。
- 3A（Arrange-Act-Assert）で整理し、1テスト1主目的を守る。
- 各テストに docstring もしくは先頭コメントで「検証する振る舞い」と「期待結果」を簡潔かつ明瞭に書く。
- Arrange / Act / Assert をコメントで区切り、構造を視覚的に追える状態に保つ。
- 変数名・関数名・コメントには、プロダクションコードや設計書で使われているユビキタス言語をそのまま使い、独自の言い換えや略語を使わない。
- 迷ったときは `references/behavior-rules.md` を参照する。

5. 言語・フレームワーク別の実践を適用する。
- 実行環境に合う参照ファイルだけを読む。
- 例: Python + pytest の場合は `references/runtime-python-pytest.md` を参照する。
- 他環境を追加する場合は `references/runtime-<lang>-<framework>.md` を増やして対応する。

6. 完了前に品質を確認する。
- まず変更範囲に絞って実行し、必要なら全体テストを実行する。
- 乱数・時刻・ネットワーク依存を制御し、毎回同じ結果になることを確認する。
- 失敗時のログだけで原因を追える状態かを確認する。

## 出力契約

このスキルを使って実装するときは、必ず次を出力する。

- 採用するテスト戦略戦略と理由（1-2文）
- 最終的なテスト配置（ディレクトリ/ファイル）
- 追加・更新したテストと対応する振る舞い
- 実行コマンドと結果要約
- 未カバーのリスクと次に書くべきテスト

## 参照ファイル

必要なものだけ読む。

- 戦略・構成: `references/strategy-and-layout.md`
- 振る舞い中心設計: `references/behavior-rules.md`
- Python + pytest プロファイル: `references/runtime-python-pytest.md`
