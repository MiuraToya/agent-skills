# 振る舞い中心テスト設計

## 1. 原則

公開契約と観測可能な振る舞いを検証する。
privateメソッド・内部状態の直接検証を主目的にしない。

## 2. テスト対象に含めるもの

- 公開関数・公開クラス・公開モジュール
- APIレスポンス、永続化結果、発行イベントなどの外部可視な結果
- 例外契約（型・メッセージ・ステータス・リトライ条件）

## 3. 避けるもの

- privateヘルパー呼び出し回数の検証を主目的にする
- 実装詳細をなぞる過剰モック
- 1テストに複数の無関係アサーションを詰め込む
- 制御不能な待機（固定sleepなど）

## 4. 内部依存テストの書き換え手順

1. 内部実装チェックを、観測可能な成果物へ置き換える。
2. 公開インターフェースから入力する。
3. 出力・副作用・契約を検証する。
4. 実装詳細の検証は補助情報に限定する。

## 5. 3A ひな型（Arrange-Act-Assert, 擬似コード）

```text
Test <condition> -> <expected_behavior>
  Arrange: preconditions and test data
  Act: invoke public API
  Assert: verify observable result
```

## 6. 例外検証ひな型（擬似コード）

```text
Test <condition> -> raises <error_type>
  Arrange: preconditions and test data
  Act: invoke public API
  Assert: verify error type and message/metadata contract
```

## 7. ドキュメンテーション規則

ブレを避けるため、フォーマットを以下に固定する。

### 7.1 検証内容と意図を docstring / コメントで明示する

docstring / コメントの役割は、**テストコード本体を読まなくても、メソッド名と説明だけで「何を検証しているか」「なぜその検証が必要か」が読み手に伝わる仕様書**として機能することにある。レビュアーやドメインに不慣れな読み手が、テスト本体を読まずに仕様を把握できる状態を目指す。

**書き方の指針**

- 基本文型: `<検証する振る舞い>のとき、<期待結果>であることを確認する。`
- 基本文型がテスト名と情報重複しても許容する。長い英語メソッド名を自然な日本語で読み下す役割を担うため、機械的に「冗長」と判断して削らない。
- 必要に応じて、業務ルール・設計意図・境界条件・防御的分岐の理由などの **WHY** を 1〜2 文添える。ドメインに不慣れな読み手でも検証意図が伝わる状態を優先する。
- 末尾は句点（`。`）で終える。
- 言語別の書き方:
  - Python: docstring（`"""..."""`）を使い、テスト関数定義の直下に置く。
  - その他言語: テスト関数定義の直下にコメントを置く（例: JS/TS なら `//`、Go なら `//`、Java なら `//`）。

**避ける書き方**

- **外面的な振る舞い（入出力の関係）の言い換えに留まる**: テスト名と docstring の両方を読んでも、「なぜこの振る舞いになっているのか」がドメインに不慣れな読み手に伝わらない状態。検証が「実装詳細をなぞっていない」ことと、説明が「外面的な振る舞いの言い換えに留まらない」ことは別問題なので分けて考える。
- 実装手順の逐次解説（内部処理を辿るような記述）。テストとして検証すべきは公開契約・観測可能な振る舞いであり、説明もそれに沿ったレベルで書く。
- 登場する用語の意味やこの分岐の存在意義が、ドメインに不慣れな読み手に全く伝わらない。

**ひな型**

```text
def test_<condition>_<expected_behavior>():
    """<検証する振る舞い>のとき、<期待結果>であることを確認する。
    （必要に応じて）<業務ルール・設計意図・境界条件などの WHY を 1〜2 文>。
    """
    ...
```

**例**

❌ 外面的な振る舞いの言い換えに留まる例（読み手は「なぜこの分岐が必要か」を知らないままになる）

```python
def test_request_with_withdrawn_user_returns_unauthorized():
    """user.status が `withdrawn` のとき、reason=`withdrawn` の Unauthorized が返ることを確認する。"""
```

✅ 業務ルール・設計意図まで踏み込んだ例（読み手は仕様の背景まで把握できる）

```python
def test_request_with_withdrawn_user_returns_unauthorized():
    """退会済みユーザからのリクエストはアクセス拒否とし、`withdrawn` 事由で Unauthorized を返すことを確認する。
    通常は認証層で除外済みだが、ドメイン層でも防御的に担保する分岐。
    """
```

### 7.2 Arrange / Act / Assert をコメントで区切る

3A の境界をコメントで明示し、テスト構造を視覚的に追えるようにする。

**フォーマット（厳守）**

- キーワードは英語の `Arrange` / `Act` / `Assert` を使う。日本語訳や略語は使わない。
- 各言語の行コメント記号 + 半角スペース1個 + キーワードのみ。末尾に句読点・補足説明は付けない（例: `# Arrange`、`// Act`）。
- セクションの直前に空行を1行入れる（先頭セクションを除く）。
- 1つの文で Act と Assert を兼ねる場合（例外検証など）は、`# Act / Assert` のようにスラッシュをスペースで挟んで連結する。
- 空になるセクション（Arrange のみ不要など）は省略可。ただし Act と Assert は必ず明示する。
- docstring の直後（Arrange があるとき）も空行を入れない。最初のセクションは docstring の直後に書く。

```text
def test_xxx():
    """<検証する振る舞い>のとき、<期待結果>であることを確認する。"""
    # Arrange
    <preconditions and test data>

    # Act
    <invoke public API>

    # Assert
    <verify observable result>
```

例外検証など Act と Assert が単一文に収まる場合:

```text
def test_xxx():
    """<検証する振る舞い>のとき、<期待結果>であることを確認する。"""
    # Act / Assert
    with pytest.raises(SomeError, match="..."):
        do_something()
```

### 7.3 ユビキタス言語を使う

テストコード内の変数名・関数名・コメント・docstring では、プロダクションコードや設計書で使われている用語をそのまま使う。
独自略語・独自パラフレーズ・テストコード固有の造語は避ける。

- 避ける: `usr`, `cust`, `tmp_record`
- 使う: `user`, `customer`, `pending_order`

ドメイン用語の不一致は、レビュー時の認知負荷とリファクタリング時の不整合の主因になる。
