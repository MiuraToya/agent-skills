# テスト戦略とディレクトリ構成

## 1. 戦略選定ルール（決定論的）

次の順序で判定する。

1. 既定値として `pyramid` を選ぶ。
2. 次をすべて満たすときのみ `diamond` を選ぶ。
- 主リスクがコンポーネント間結合・契約不整合にある。
- E2E/UIフロー数が少なく安定している。
- 高速フィードバックを最優先する。

混在する場合は `pyramid` を維持し、リスク境界だけ統合テストを厚くする。

## 2. 目安比率

初期値として使い、欠陥履歴で調整する。

- Pyramid
- Unit: 60-75%
- Integration/Contract/API: 20-35%
- E2E/UI: 5-10%

- Diamond
- Unit: 20-35%
- Integration/Contract/API: 50-70%
- E2E/UI: 10-20%

## 3. 推奨ディレクトリ構成（汎用）

```text
project/
  src/
    ...
  tests/
    unit/
      <test files>
    integration/
      <test files>
    contract/
      <test files>
    e2e/
      <test files>
    fixtures/
      <shared fixtures>
```

言語ごとの慣習がある場合は、実行環境プロファイル側で上書きする。

## 4. レイヤー別の書き方

- Unit
- 実ネットワーク/実DBなどの外部副作用を使わない。
- テストダブルで依存を隔離する。

- Integration/Contract
- Integrationは「複数コンポーネントを実際につないで検証するテスト」を指す。
- Contractは「境界のI/F契約（スキーマ、ステータス、エラー形式など）の互換性検証」を指す。
- 運用上は `integration` にまとめてもよい。境界互換性を明示したい場合のみ `contract` を分ける。
- DB/API/キュー/外部サービス境界の整合を確認する。
- シリアライズ、スキーマ、エラー経路を検証する。
- DBは重要経路で実DB（例: MySQLコンテナ）を優先し、外部クラウド依存はエミュレータ/フェイク（例: moto）を併用する。

- E2E
- 本番影響の大きいクリティカルフローに限定する。
- 内部遷移ではなく利用者視点の結果を検証する。

## 5. 命名規則

- ファイル: 実行環境の慣習に合わせる。
- テストケース: `<条件>_<期待結果>` で意味が読める名前にする。
- 1テスト1主目的を維持する。
