# PR Review Response Workflow

## 概要

PR レビューの取りこぼしと対応漏れを防ぐため、コメントごとに「対応 → 返信」を紐づけて進める。次の3点を必ず守る。

- 合意なしに着手しない。
- すべてのレビューコメントは「修正する / 修正しない / 要相談 / ユーザー返信」のいずれかに分類して扱う。
- スキルから返信するコメントには、対応したコミットハッシュ（または合意した不対応理由）を必ず添える。

## 前提

- `gh` (GitHub CLI) が認証済みで利用できる。
- 対象 PR の番号 (または URL) を入力として受け取る。
- 作業ブランチに対して `git push` 権限がある。

## 実行ワークフロー

### 1. レビュー内容を取得して整理する

次のコマンドでレビュー情報を取得する。

- `gh pr view <PR>` — PR の概要・タイトル・ブランチ
- `gh api repos/<owner>/<repo>/pulls/<PR>/comments` — インラインレビューコメント
- `gh api repos/<owner>/<repo>/pulls/<PR>/reviews` — 全体レビュー (approve / request changes 本文)

取得結果を次の表形式に整理する。スレッド返信が既にあるコメントは「対応済み候補」としてマークし、ユーザーに確認する。

| コメントID | ファイル:行 | 指摘要約 | 既存返信 | 状態 |
|---|---|---|---|---|

### 2. 対応方針を立てて、ユーザーに提案する

各コメントを次の4カテゴリに分類する。

- **修正する** — どのファイルをどう変えるかを1-2文で書く。
- **修正しない** — 理由（設計意図・スコープ外・別 PR で対応など）を1-2文で書く。
- **要相談** — 判断できない理由と、ユーザーへの質問事項を書く。
- **ユーザー返信** — ユーザーが自分で返信するためスキルから返信不要、と明示的に指示されたコメント。スキルからの修正・返信は一切行わない。

「修正する」のうち、関連性がある複数コメントは1コミットへまとめる候補としてグルーピング案を示す。グルーピングの基準は以下とする。

- 同一ファイル・同一関心事 (同じバグ・同じリファクタ目的) を扱うコメント。
- 一方を直すと他方も自然に解消されるコメント。

分類とグルーピング案をまとめてユーザーに提示し、**合意を得るまで着手しない**。「修正しない」コメントについては、ユーザーと合意した返信文面の原型もこの時点で固める。

### 3. 合意済みの修正を順にコミットする

- 既定: 1コメント = 1コミット。
- Step 2 で合意したグルーピングがあるコメント群のみ、まとめて1コミットにする。
- コミットメッセージは次の形式とする。

```text
<type>: <要約>

Refs: PR #<PR番号> review comment <コメントID>[, <コメントID>...]
```

各コミットを作成したら、コミットハッシュをコメントID との対応表に記録する。

### 4. 各コメントへ返信する

返信は次のコマンドで投稿する。

```text
gh api -X POST repos/<owner>/<repo>/pulls/<PR>/comments/<コメントID>/replies -f body=<返信本文>
```

- **修正したコメント** には次の形式で返信する。

  ```text
  <対応内容の要約>
  commit: <コミットハッシュ>
  ```

  対応内容の要約は基本 1 行で書く。1 行に収まらない場合は、簡潔な箇条書きにしてもよい。
  いずれの場合も、レビュアーがコミットを開かずに変更の趣旨を把握できる粒度を保つ。

  例 (1 行):

  ```text
  引数の null チェックを追加しました。
  commit: a6b94a5
  ```

  例 (箇条書き):

  ```text
  - 引数の null チェックを追加
  - 該当ケースの単体テストを追加
  commit: a6b94a5
  ```

  グルーピングで複数コメントが同一コミットを共有する場合、各コメントへ同じハッシュを返信する（要約はコメントごとの観点に合わせて書き分けてよい）。

- **修正しないコメント** には、Step 2 でユーザーと合意した理由を返信する。形式の例:

  ```text
  <合意した理由> という理由から、本 PR では対応不要と判断しました。
  ```

  単に「対応しません」だけにせず、必ず合意済みの理由を含める。

- **ユーザー返信カテゴリのコメント** には、スキルから返信を投稿しない。最終報告 (Step 5) で「ユーザー返信予定」として明示するに留める。

### 5. push と最終報告

- 作業ブランチを `git push` する。
- 次を報告する。
  - 修正したコメント数 / 修正しないと判断したコメント数 / 残った要相談コメント数
  - 作成したコミット数とハッシュ一覧
  - 各コメントへの返信投稿が完了したかどうか
  - 別 PR で対応するとしたコメントがある場合はその一覧

## 出力契約

このスキルを使うときは、必ず次を出力する。

- レビューコメント一覧と分類・グルーピング案 (Step 2 合意確認用)
- 作成したコミットとコメントID の対応表 (Step 3 後)
- 各コメントへの返信本文 (Step 4 後)
- 未対応・要相談として残ったコメント、もしくは別 PR で扱うコメントの一覧

## やってはいけないこと

- ユーザー合意前にファイル修正やコミット作成を始める。
- 1つのコミットに、関連性のない複数コメントの修正を詰め込む。
- コミットハッシュなしで返信する。または対応内容の要約を省いてハッシュだけを返信する。
- 修正しないコメントを未返信のまま放置する（ただし「ユーザー返信」カテゴリとして合意済みのものを除く）。
- 「ユーザー返信」と明示されたコメントに対してスキルから勝手に返信を投稿する。
- レビュー以外の自発的なリファクタや無関係な修正を同じ PR ブランチに混ぜる。
